// Acto 1
Antes de tocar algo, leer y entender
Aprender a leer un unit file de systemd usando cron como ejemplo
¿Por qué leer un unit file antes de operar un servicio?
▶

El unit file es el contrato del servicio

Cuando un servicio falla o se comporta de manera inesperada, lo primero que hace un administrador experimentado no es reiniciarlo — es leer su unit file. Ahí está toda la información sobre cómo ese servicio fue diseñado para funcionar.

Un unit file te dice: qué comando ejecuta el servicio, de qué otros servicios depende, qué pasa si se cae (¿se reinicia solo?), y en qué momento del arranque se activa.

¿Por qué cron como ejemplo?

cron es el programador de tareas de Linux — ejecuta comandos automáticamente en horarios definidos. En un servidor de producción es fundamental: hace backups, limpia logs, ejecuta scripts de mantenimiento. Si cron falla silenciosamente y nadie lo nota, los backups dejan de hacerse.

🏭
Caso real: En un servidor de producción, el equipo detectó que los backups nocturnos no se ejecutaban desde hacía una semana. Al revisar con systemctl cat cron.service, encontraron que cron tenía configurado After=network.target pero la red tardaba más de lo esperado en iniciar, haciendo que cron arrancara antes de que la red estuviera lista y fallara silenciosamente. Leer el unit file fue el primer paso para diagnosticar el problema.

Qué vas a encontrar en un unit file

DirectivaSección¿Qué significa?
Description=[Unit]Descripción legible del servicio
After=[Unit]Qué debe estar iniciado ANTES que este servicio
Requires= / Wants=[Unit]Dependencias duras vs blandas. Si falla un Requires, este servicio también falla.
ExecStart=[Service]El comando exacto que ejecuta el servicio
Restart=[Service]Política de reinicio automático si el servicio se cae
WantedBy=[Install]En qué target se activa este servicio al arrancar
💡
La diferencia entre Requires= y Wants= es crítica: si A tiene Requires=B y B falla, A también falla. Si tiene Wants=B, A intenta arrancar igual aunque B no esté disponible.

// Acto 2
Antes de detener algo, saber qué arrastra
Listar dependencias de un servicio con systemctl
¿Por qué importa conocer las dependencias?
▶

El efecto dominó de los servicios

En un servidor real, los servicios no viven solos. Un servidor web puede depender de una base de datos, que depende de la red, que depende de interfaces de red configuradas. Si detenés un servicio sin conocer su árbol de dependencias, podés tirar en cadena cosas que no tenías en mente.

🏭
Caso real: Un administrador necesitaba reiniciar el servicio de red en un servidor con varios contenedores Docker corriendo. Sin revisar las dependencias, al bajar la red, Docker perdió conectividad y mató automáticamente todos los contenedores activos que tenían política de restart: always, generando un reinicio masivo no planificado de todos los servicios. Verificar dependencias antes hubiera permitido planificar una ventana de mantenimiento.

Dependencias en dos sentidos

systemd maneja dependencias en ambas direcciones. Antes de detener un servicio, un buen administrador verifica:

DirecciónComando¿Qué muestra?
Lo que necesitasystemctl list-dependencies sshQué debe estar corriendo para que SSH funcione
Lo que depende de élsystemctl list-dependencies --reverse sshQué se cae si detenés SSH

// Acto 3
Operar con criterio — start, stop, enable, disable
La diferencia entre lo que pasa ahora y lo que pasa siempre
start/stop vs enable/disable — el error más común
▶

La confusión que tira servidores en producción

Esta es una de las confusiones más frecuentes y costosas en administración de sistemas. Muchos administradores junior configuran un servicio, lo inician con start, creen que ya está listo para producción, y al primer reinicio del servidor el servicio no arranca.

Comando¿Qué hace?¿Persiste al reiniciar?
systemctl startInicia el servicio ahora mismo❌ No — si reinicias, no arranca
systemctl stopDetiene el servicio ahora mismo❌ No — si reinicias, vuelve a arrancar
systemctl enableCrea el enlace simbólico para arranque automático✅ Sí — arranca en cada boot
systemctl disableElimina el enlace simbólico✅ Sí — no arranca en el próximo boot
⚠️
Importante: enable y disable NO inician ni detienen el servicio ahora mismo — solo afectan el comportamiento en el próximo arranque. Para hacer ambas cosas a la vez usás systemctl enable --now o systemctl disable --now.
🏭
Caso real: Tras configurar un servidor web nginx, el administrador ejecutó systemctl start nginx y verificó que funcionaba. El servidor pasó a producción. Tres días después hubo un corte de luz breve. Al volver, el servidor web no respondía — nginx nunca había sido habilitado con enable, así que no arrancó automáticamente. El sitio estuvo caído hasta que alguien se dio cuenta y lo inició manualmente.

Conocer el estado del sistema — targets
Entender en qué "modo" está corriendo el servidor y qué implica cada nivel
¿Qué es un target y por qué importa?
▶

Los targets son los "niveles de funcionamiento" del servidor

Un target en systemd es un estado del sistema: define qué servicios deben estar corriendo y cuáles no. Es el equivalente moderno de los "runlevels" de SysVInit. Saber en qué target está tu servidor te dice qué capacidades tiene disponibles en ese momento.

La jerarquía de targets — de más a menos servicios

graphical.target
Todo activo + posibilidad de interfaz gráfica. En Ubuntu Server sin GUI instalada, funciona igual que multi-user.target en la práctica, pero systemd lo reporta como default porque es el nivel "techo" definido.
multi-user.target
Red activa, multiusuario, todos los servicios de sistema. El modo normal de un servidor Linux sin entorno gráfico.
rescue.target
Monousuario. Sin red. Sin servicios multiusuario. Solo root puede operar. Sistema de archivos montado en lectura/escritura. Para mantenimiento de emergencia.
emergency.target
Mínimo absoluto. Sistema de archivos raíz en solo lectura. Sin casi nada activo. Para cuando el sistema está tan dañado que ni rescue arranca correctamente.
💡
¿Por qué Ubuntu Server muestra graphical.target como default? No significa que tenga GUI — significa que ese es el nivel máximo al que aspira el sistema. Si no hay entorno gráfico instalado, systemd llega hasta donde puede (multi-user) y ahí se queda. Es confuso por el nombre, pero es correcto.

El último recurso — modo rescate
rescue.target: cuándo usarlo, qué hace y qué no hace
rescue.target en la vida real de un administrador
▶

El modo rescate no es para curiosos — es para emergencias

Un administrador que nunca usó rescue.target es un administrador que todavía no tuvo su primera noche de pánico frente a un servidor que no arranca. Este target existe para situaciones donde el sistema está parcialmente roto y necesitás operar en él con la menor cantidad de "ruido" posible.

Situaciones reales donde rescue.target salva la noche

Situación¿Por qué rescue.target?
Error en /etc/fstabSi una entrada mal escrita en fstab impide el montaje de particiones, el sistema puede no arrancar normalmente. En rescue podés corregirlo con la raíz montada en escritura.
Servicio que rompe el bootUn servicio mal configurado puede hacer que el arranque normal se congele. En rescue ese servicio no arranca y podés deshabilitarlo o corregirlo.
Password de root perdidaCon acceso físico, se puede arrancar en rescue para cambiar la contraseña de root sin conocer la actual.
Corrección de permisos críticosSi los permisos de archivos del sistema se corrompieron, en rescue podés repararlos sin que otros procesos interfieran.
Mantenimiento de discoPara correr fsck (verificación del sistema de archivos) en una partición, ésta no puede estar en uso. En rescue podés desmontar particiones que normalmente siempre están montadas.
⚠️
Sobre las conexiones activas: Al ejecutar systemctl isolate rescue.target, systemd detiene los servicios multiusuario — incluyendo SSH. Las conexiones SSH activas pueden sobrevivir si su proceso hijo ya estaba corriendo (el kernel no cierra conexiones TCP establecidas cuando para el demonio padre). Sin embargo, las conexiones de otros usuarios sí deberían cortarse. La tuya puede sobrevivir porque sos vos quien ejecutó el aislamiento. En producción, siempre avisá antes de aislar.
💡
rescue vs emergency: Si rescue.target no arranca porque el problema es más profundo (sistema de archivos corrupto, problemas de kernel), se usa emergency.target, que monta la raíz en solo lectura y no arranca casi nada. Es el último escalón antes de arrancar desde un live USB.
